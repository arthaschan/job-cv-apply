# 岗位扫货文档管理 — 文件组织与合并工作流

> 当用户说"把V3更新进去"/"合并到LinkedIn和JobsDB文档"/"删掉这个核实版"时，执行此流程。

## 文件组织

### 存放位置
所有岗位扫货文档在 `~/job-search/inbox/`（快速交付物，不放知识库）。

### 命名规范
```
{平台}岗位扫货_{YYYYMMDD}.md     ← 平台原始扫描
Gmail岗位扫货_{YYYYMMDD}.md      ← 邮箱扫描报告
岗位扫货_V{N}_{描述}_{YYYYMMDD}.md  ← 合并核实/评审文档（中间产物，合并后删除）
```

### 当前文件清单（示例）
```
~/job-search/inbox/LinkedIn岗位扫货_20260603.md   ← LinkedIn源
~/job-search/inbox/JobsDB岗位扫货_20260605.md     ← JobsDB源
~/job-search/inbox/Gmail岗位扫货_20260608.md      ← Gmail邮箱扫描
~/job-search/inbox/岗位扫货_V3_人工核实版_20260605.md  ← 合并评审（临时）
```

**关键规则**：
- 平台文档按**来源**分（LinkedIn / JobsDB / Gmail），不是按日期分
- 同一平台有多次扫描时，追加新section到现有文件（不覆盖历史数据）
- 合并评审文档是**临时中间产物**，合并完成后必须删除

## 合并核实文档到平台文档（核心流程）

当用户创建了一个合并评审文档（如V3人工核实版），需要将其中的变更拆分回各平台源文档：

### Step 1: 读取并分类
1. 读取合并评审文档
2. 识别每条记录的**来源平台**（文档中通常有"来源"列标注 LinkedIn / JobsDB）
3. 分类为：LinkedIn来源 vs JobsDB来源

### Step 2: 读取平台源文档
1. 读取对应的平台文档（搜索 `~/job-search/inbox/{平台}岗位扫货_*.md`）
2. 理解现有结构（强推/观望/避坑等分区）

### Step 3: 逐条更新平台文档
对合并评审中的每条记录，根据V3结论更新平台文档：

**状态变更类型**：
| 变更类型 | 操作 | 示例 |
|----------|------|------|
| ↓降级（强推→避坑） | 修改状态列为🚫避坑，更新备注 | Warner Music: 待投 → 🚫 避坑 |
| ↓降级（强推→待确认） | 修改匹配度星数，加❓前缀 | 中国ICT大厂: ⭐⭐⭐⭐ → ⭐⭐⭐ ❓待确认 |
| ↑升级（观望→立即投） | 修改状态列为🎯**立即投** | 某科技公司: 待投 → 🎯立即投 |
| =信息更新 | 更新公司名/备注 | OCFT: 待确认 → OCFT（公司名已确认） |
| ↓降级（强推→观望） | 降星数，加❓ | 某公司: ⭐⭐⭐⭐ → ⭐⭐⭐ ❓ |

**Patch模式**（用 `patch` 工具，不用 `write_file` 覆盖）：
```
@@ 表格行的唯一标识文本 @@
-| 原状态行 |
+| 新状态行 |
```
- `old_string` 要包含足够的上下文确保唯一匹配（通常用公司名+岗位名片段）
- 一次 patch 调用可以包含多个 `@@` hunk

### Step 4: 添加变更汇总表
在每个平台文档末尾追加一个"V3人工核实更新"section：

```markdown
---

## 📋 V3 人工核实更新（YYYY-MM-DD → YYYY-MM-DD 合并）

> 以下为V3核实版对{平台}来源岗位的状态变更汇总

| 变更 | 岗位 | 公司 | 原状态 | V3结论 |
|------|------|------|--------|--------|
| ↓降级 | ... | ... | 强推⭐⭐⭐⭐ | 🚫避坑：原因 |
| ↑升级 | ... | ... | 观望⭐⭐⭐ | 🎯**立即投**（原因）|
```

### Step 5: 更新下一步建议
如果平台文档有"下一步建议"section，更新已确认的条目（划掉已完成的，添加新建议）。

### Step 6: 删除合并评审文档
```bash
rm "~/job-search/inbox/岗位扫货_V{N}_{描述}_{YYYYMMDD}.md"
```

## Pitfall

### P1: 来源标注可能不准
合并评审文档中的"来源"列可能有误（如某研究院实际来自JobsDB但标为LinkedIn）。以**平台源文档中是否存在该岗位**为准，不要盲信合并文档的来源标注。

### P2: 不要覆盖历史数据
平台文档可能有多个时间点的section（首轮+第二轮）。用 `patch` 做精确替换，不要用 `write_file` 覆盖整个文件。

### P3: 汇总表要独立于正文
变更汇总表放在文档末尾作为独立section，不要混入正文的强推/观望/避坑分区。这样用户可以一眼看到"V3改了什么"。

### P4: 同一岗位可能出现在多个平台
如果一个岗位同时出现在LinkedIn和JobsDB的扫描中，按合并评审文档标注的主要来源更新，不要重复更新两个文件。

### P5: 避坑岗位的链接保留
降级为避坑的岗位，保留原有的链接/URL，方便用户后续复查。只改状态和备注，不删链接。
